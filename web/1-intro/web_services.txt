Rust для веб-сервисов
Стандартная библиотека Rust реализует TCP, можно открыть сокет TCP (или UDP-сокет) и прослушивать входящие сообщения.
Однако реализации HTTP нет. Поэтому,используют одну из сторонних библиотек и пишут с HTTP c нуля.
Веб-фреймворк Actix Web использует собственную реализацию HTTP-сервера actix-server.
При использовании Warp, Axum или Rocket все они используют Hyper в качестве веб-сервера (открытие сокета, ожидание и анализ HTTP-сообщений).

Язык программирования предлагают следующие концепции, позволяющие создавать приложения, способные выполнять работу асинхронно:
- Syntax помечает фрагмент кода как асинхронный
- Type может сохранять состояние асинхронного выполнения
- Thread scheduler (runtime) способ управления потоками или другими методами, чтобы перевести работу в фоновый режим и обеспечить её выполнение
- Kernel abstractions использование асинхронных методов ядра в фоновом режиме
---
Tokio — это асинхронная среда выполнения (или планировщик потоков). Сам Tokio использует (через другой ящик, называемый Mio) асинхронный интерфейс прикладного программирования ядра (API)
из операционной системы.
Вначале аннотируем функцию с помощью #[tokio::main].
Далее можем пометить функцию как асинхронную с помощью стандартного синтаксиса Rust и ожидать
типа future. 
Future в Rust — это трейт, который может быть реализован для типов. Этот трейт определяет, что реализация должна иметь тип Output (указывающий, что возвращает future после завершения вычисления), и функцию, называемую poll (которую среда выполнения может вызвать для работы с future).
