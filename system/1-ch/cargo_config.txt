Указание выходных целевых объектов для пакета:
• [[bin]]: Двоичный целевой объект — это исполняемая программа, которую можно запустить после сборки.
• [lib]: Библиотечное целевое объект создаёт библиотеку, которую могут использовать другие библиотеки и исполняемые файлы.
• [[example]]: Этот целевой объект полезен библиотекам для демонстрации использования внешних API пользователям с помощью примера кода. Исходный код примера, расположенный в
каталоге example, может быть собран в исполняемые двоичные файлы с помощью этого целевого объекта.
• [[test]]: Файлы, расположенные в каталоге tests, представляют собой интеграционные тесты, и каждый из них может быть скомпилирован в отдельный исполняемый двоичный файл.
• [[bench]]: Тестовые функции, определённые в библиотеках и двоичных файлах, компилируются в отдельные исполняемые файлы.
---
Указание зависимостей для пакета:
• [dependencies]: Библиотечные или двоичные зависимости пакета
• [dev-dependencies]: Зависимости для примеров, тестов и бенчмарков
• [build-dependencies]: Зависимости для скриптов сборки (если таковые указаны)
• [target]: Предназначено для кросс-компиляции кода для различных целевых архитектур
---
Указание профилей сборки:
• dev: Команда сборки Cargo по умолчанию использует профиль dev. Пакеты, собранные с этой опцией, оптимизированы для скорости компиляции.
• release: Команда сборки Cargo –-release включает профиль выпуска, подходящий для выпуска в продакшн и оптимизированный для скорости выполнения.
• test: Команда сборки Cargo использует этот профиль. Он используется для сборки тестовых исполняемых файлов.
• bench: Команда сборки Cargo создает исполняемый файл бенчмарка,
---
Рабочее пространство — это единица организации, в которой несколько пакетов могут быть сгруппированы в проект. Это полезно для экономии места на диске
и времени компиляции при наличии общих зависимостей между набором связанных пакетов.
Раздел [workspace] можно использовать для определения списка пакетов, входящих в
рабочее пространство.
