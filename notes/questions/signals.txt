Отлично, это уже **уровень ОС + Rust**, тут реально важно понять *что где живёт*.
Пойду по порядку: **что такое сигналы → какие реально нужны → как Rust с ними работает → как запретить потоку их получать**.

---

# 1. Что такое сигналы (по-взрослому)

**Сигнал** — это **асинхронное уведомление от ядра процессу**.

Важно:

* сигнал **приходит процессу**
* **доставляется одному из потоков**
* *какому именно* — зависит от масок и состояния

Это не callback и не interrupt в коде.
Это ближе к:

> «эй, процесс, с тобой что-то случилось»

---

## Откуда приходят сигналы

* 🧠 **ядро**

  * `SIGSEGV` — invalid memory
  * `SIGPIPE` — write в закрытый сокет
* ⌨️ **терминал**

  * `Ctrl+C` → `SIGINT`
  * `Ctrl+\` → `SIGQUIT`
* 🧑‍💻 **другой процесс**

  * `kill(pid, SIGTERM)`
* ⏱️ **таймеры**

  * `SIGALRM`

---

# 2. Самые нужные сигналы (90% случаев)

Вот **практический минимум**, без экзотики:

| Сигнал    | Когда                  | Что обычно делают |
| --------- | ---------------------- | ----------------- |
| `SIGINT`  | Ctrl+C                 | graceful shutdown |
| `SIGTERM` | kill / systemd         | graceful shutdown |
| `SIGQUIT` | Ctrl+\                 | core dump + exit  |
| `SIGHUP`  | терминал умер / reload | перечитать конфиг |
| `SIGSEGV` | segfault               | crash             |
| `SIGABRT` | abort                  | crash             |
| `SIGPIPE` | write в закрытый pipe  | игнорируют        |

📌 **`SIGKILL` и `SIGSTOP`**

* нельзя перехватить
* нельзя игнорировать
* ядро убьёт процесс

---

# 3. Очень важное правило POSIX

> **Сигналы — это уровень ПРОЦЕССА,
> но маски — уровень ПОТОКА**

То есть:

* сигнал адресован процессу
* доставляется *одному* потоку
* **поток может сказать: “я такие сигналы не принимаю”**

---

# 4. Как ядро выбирает поток

Упрощённо:

1. смотрит, какие потоки **не блокируют сигнал**
2. выбирает любой подходящий
3. если все блокируют → сигнал pending

👉 **Да, “любой поток” — это нормально**

---

# 5. Как Rust работает с сигналами

## Коротко

> **Rust сам сигналы НЕ обрабатывает**

`std::thread`, `std::sync`:

* вообще не знают о сигналах
* не ставят handlers
* не блокируют

Rust = пользовательский код
Сигналы = ОС

---

## Что происходит по умолчанию

| Сигнал    | Поведение           |
| --------- | ------------------- |
| `SIGINT`  | завершение процесса |
| `SIGTERM` | завершение процесса |
| `SIGPIPE` | **terminate**       |
| `SIGSEGV` | crash               |

---

# 6. Почему нельзя писать signal handler на Rust напрямую

POSIX требует:

> signal handler должен быть **async-signal-safe**

В signal handler **НЕЛЬЗЯ**:

* аллокацию
* mutex
* println
* большинство libc
* вообще почти всё

Rust:

* аллокирует
* использует TLS
* вызывает runtime

👉 **поэтому “чистый” handler на Rust — UB**

---

# 7. Правильный паттерн: self-pipe / signalfd

### В Linux — **signalfd** (лучший вариант)

Идея:

* сигналы → **файловый дескриптор**
* читаешь их как события
* НИКАКИХ async handlers

---

# 8. Как это делают в Rust (правильно)

Через crates:

* `signal-hook`
* `nix`

### Пример с `signal-hook`

```rust
use signal_hook::consts::signal::*;
use signal_hook::iterator::Signals;

fn main() -> std::io::Result<()> {
    let mut signals = Signals::new([SIGINT, SIGTERM])?;

    for sig in signals.forever() {
        match sig {
            SIGINT | SIGTERM => {
                println!("graceful shutdown");
                break;
            }
            _ => {}
        }
    }

    Ok(())
}
```

📌 Здесь:

* используется self-pipe
* безопасно
* работает с потоками

# 9. Теперь главный вопрос:

## 🔥 как запретить потоку принимать сигналы

Ты абсолютно прав:

> *«любой поток из процесса и неизвестно какой»*

### Решение: **маски сигналов (pthread_sigmask)**

---

## 10. Блокировка сигналов в потоке (Linux / POSIX)

Через `nix`:

```rust
use nix::sys::signal::{sigprocmask, SigSet, SigmaskHow, Signal};

fn block_signals() {
    let mut set = SigSet::empty();
    set.add(Signal::SIGINT);
    set.add(Signal::SIGTERM);

    sigprocmask(SigmaskHow::SIG_BLOCK, Some(&set), None).unwrap();
}
```

📌 ВАЖНО:

* это **действует ТОЛЬКО для текущего потока**
* новые потоки **наследуют маску**

---

## 11. Канонический паттерн серверов

### Шаги:

1. **В main потоке**

   * блокируем сигналы
2. **Создаём worker threads**

   * они наследуют блокировку
3. **Отдельный signal thread**

   * разблокирует сигналы
   * обрабатывает их

---

### Пример архитектуры

```text
main thread
 ├─ block SIGINT/SIGTERM
 ├─ spawn workers (blocked)
 └─ spawn signal thread (unblocked)
```

---

### Код-скелет

```rust
fn main() {
    block_signals(); // до spawn!

    // workers
    for _ in 0..4 {
        std::thread::spawn(|| worker());
    }

    // signal thread
    std::thread::spawn(|| signal_loop());

    // join...
}
```

---

## 12. Почему это ПРАВИЛЬНО

* сигналы **никогда** не прилетят в worker
* вся логика shutdown — в одном месте
* нет гонок
* нет UB

Так делают:

* nginx
* systemd
* postgres
* JVM

---

# 13. Почему Rust сам это не скрывает

Потому что:

* сигналы — **часть ABI ОС**
* разные платформы
* разные semantics

Rust принципиально:

> *«не прячет ОС под ковёр»*

---

# 14. Ответы на частые мифы

❌ «signal handler можно писать на Rust»
✅ только через pipe / signalfd

❌ «SIGINT приходит в main thread»
✅ **нет**, куда угодно

❌ «Mutex в signal handler — ок»
✅ **катастрофа**

---

# 15. Ментальная модель (запомни)

```text
Сигнал → процесс
Маска → поток
Обработка → отдельный поток
```
