Коротко:
`dyn` — это **динамическая диспетчеризация через trait-объект**.
А теперь разложим по косточкам, без магии.

---

## Вся конструкция целиком

```rust
Box<dyn Error>
```

Читается как:

> «Box с объектом *какого-то типа*, который **реализует trait `Error`**, но **конкретный тип мне сейчас не важен**»

---

## Зачем тут вообще `dyn`

В Rust **trait сам по себе — не тип**.

```rust
Error        // ❌ не тип
dyn Error    // ✅ trait object
```

`dyn` говорит компилятору:

> «Я хочу работать **не со статическим типом**, а через **vtable**»

То есть:

* **какой конкретно error — решится в runtime**
* вызовы методов идут через таблицу виртуальных функций

---

## Почему без `Box` нельзя

Trait object — **unsized тип** (`?Sized`):

```rust
dyn Error // ❌ неизвестен размер
```

А `Box<T>` умеет хранить `?Sized`:

```rust
Box<dyn Error> // ✅ размер Box известен (указатель)
```

Фактически в памяти:

```
Box
 ├─ ptr на данные
 └─ ptr на vtable
```

---

## Что внутри `dyn Error`

Trait `Error` выглядит (упрощённо) так:

```rust
pub trait Error: Debug + Display {
    fn source(&self) -> Option<&(dyn Error + 'static)> { ... }
}
```

Поэтому любой из этих типов подходит:

```rust
std::io::Error
std::num::ParseIntError
anyhow::Error
твой_кастомный_тип
```

---

## Пример без `dyn` — почему не работает

```rust
fn f() -> Error { ... } // ❌
```

Компилятор:

> я не знаю, сколько байт возвращать

---

## Правильный вариант

```rust
fn f() -> Box<dyn Error> {
    Box::new(std::io::Error::new(
        std::io::ErrorKind::Other,
        "oops",
    ))
}
```

Или чаще:

```rust
fn f() -> Result<(), Box<dyn Error>> {
    Ok(())
}
```

---

## Что реально происходит при вызове метода

```rust
err.to_string()
```

1. Rust берёт vtable из `dyn Error`
2. Находит нужную реализацию `Display`
3. Вызывает её

Это **медленнее**, чем generics, но:

* разница микроскопическая
* гибкость огромная

---

## Сравнение с generics (важно понять)

### Статически (без `dyn`)

```rust
fn f<E: Error>(e: E) { }
```

* тип известен на этапе компиляции
* monomorphization
* быстрее
* **нельзя хранить разные типы в одном месте**

---

### Динамически (`dyn`)

```rust
fn f(e: Box<dyn Error>) { }
```

* тип неизвестен до runtime
* vtable
* можно вернуть **любой** error
* идеально для API

---

## Почему `dyn` пишут явно (начиная с Rust 2018)

Раньше было:

```rust
Box<Error>
```

Теперь **запрещено**, чтобы ты всегда понимал:

> ⚠️ тут динамическая диспетчеризация

---

## Когда `Box<dyn Error>` — правильный выбор

✔️ публичные API
✔️ `main() -> Result<(), Box<dyn Error>>`
✔️ когда тип ошибки не важен
✔️ быстро собрать прототип

Когда **не надо**:

❌ горячие циклы
❌ embedded / no_std
❌ если набор ошибок фиксирован → `enum`

---

## Мини-ментальная модель

> `dyn Trait` = «интерфейс + vtable»
> `Box<dyn Trait>` = «указатель на что угодно, что реализует Trait»
