11. Автоматизированные тесты
==========================
В самом простом случае тест — это функция, которая аннотируется атрибутом test
```src/lib.rs
#[cfg(test)]
mod tests {
  #[test]
  fn it_works() {
    assert_eq!(2 + 2, 4);
  }

  #[test]
  fn exploration() {
    assert_eq!(2 + 2, 4);
  }

  #[test]
  fn another() {
    panic!("Сделать этот тест неуспешным");
  }
}
```
---
Контроль выполнения тестов
cargo test компилирует код в тестовом режи­ме и выполняет результирующий двоичный тест
cargo test --help, увидеть аргументы, которые можно использовать с командой cargo test
cargo test -- --help, увидеть аргументы, которые можно использовать после разделителя --
---
Параллельное и последовательное выполнение тестов
По умолчанию тесты выполняются парал­лельно с использованием потоков исполнения. при парелельном выполнении они не должны зависеть друг от друга или от какого-либо совместного состояния
Для контролорля числа потоков
cargo test -- --test-threads=1
* установили число тестовых потоков исполнения равным 1, сказав программе не использовать никакого параллелизма
Выполнение тестов с одной нитью ис­полнения займет больше времени, чем их параллельное выполнение, но тесты не будут мешать друг другу, если они делят между собой совместное состояние
---
По умолчанию, при тестах с println. мы увидим вывод текста в отчете о тестак
fn prints_and_returns_10(a: i32) -> i32 {
  println!("Я получил значение {}", a);
  10
}
что бы не видеть, то что печатается
cargo test -- --nocapture
---
Выполнение одиночных тестов
cargo test one_hundred
Выполнение подмножества тестов по имени
cargo test add
running 2 tests
test tests::add_two_and_two ... ok
test tests::add_three_and_two ... ok

Игнорирование нескольких тестов
Аннотация
#[test]
#[ignore]
выполнить игнорируемые
cargo test -- --ignored
===
Организация тестов
---
Модульные тесты малы и более сфокусиро­ваны на проверке одного модуля в отдельности, они могут тестировать приватныеинтерфейсы.
В каждом файле принято создавать модуль с именем tests, содержащим тестовые функции, и ан­нотировать модуль с помощью cfg(test).
ннотация #[cfg(test)] для модуля tests говорит компилятору выполнить те­
стовый код, только при запуске команды cargo test, а не при запуске cargo build.
---
Интеграционные тесты полностью внешние по отношению к библиоте­ке и используют код так же, как и любой другой внешний код, применяя только публичный интерфейс и потенциально проверяя несколько модулей в одном тесте
Принято создавать каталог tests на верхнем уровне каталога проекта, рядом с src
```tests/integration_test.rs
use adder;

  #[test]
  fn it_adds_two() {
    assert_eq!(4, adder::add_two(2));
  }
```
сargo test
cargo test --test integration_test
---
Подмодули в интеграционных тестах
альтернативное соглашение об именовании
tests/common/mod.rs, можно использовать из любого файла интеграционного теста в качестве модуля:
```tests/common/mod.rs
pub fn setup() {
  // код функции setup, характерный для тестов вашей библиотеки,
  // будет находиться здесь
}
```
теперьможно
```tests/integration_test.rs
use adder;
mod common;
  #[test]
  fn it_adds_two() {
    common::setup();
    assert_eq!(4, adder::add_two(2));
  }
```
---
Интеграционные тесты для двоичных упаковок
Если проект представляет собой двоичную упаковку, содержащую только файл src/main.rs, и в нем нет файла src/lib.rs, то мы нельзя создавать интеграционные тесты в каталоге tests и вводить функции, определенные в файле src/main.rs, в область видимости с помощью инструкции use
