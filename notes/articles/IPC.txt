===
Rust IPC from Scratch: Mmap, Atomics, and Memory Layouts
Межпроцессное взаимодействие на Rust с нуля: Mmap, атомарные операции и организация памяти.
Migo Lee
===
Когда требуется молниеносная межпроцессная связь (IPC), стандартные сокеты или каналы иногда могут стать узким местом из-за постоянного переключения контекста и копирования данных между пользовательским пространством и пространством ядра.

Одним из решений являются файлы, отображаемые в память (mmap), поддерживаемые разделяемой памятью POSIX. Отображая сегмент разделяемой памяти непосредственно в адресное пространство нескольких процессов, производители и потребители могут обмениваться данными со скоростью оперативной памяти — копирование не требуется.

Прежде чем писать код, стоит задать фундаментальный вопрос: зачем создавать многопроцессные IPC, если существует стандартная многопоточность? Если вам нужна только параллельная обработка, создание потоков внутри одного исполняемого файла Rust намного проще. Однако переход к многопроцессной архитектуре с разделяемой памятью дает ряд преимуществ, таких как:

Изоляция ошибок: даже если один процесс аварийно завершается, остальные продолжают работать без сбоев.

Развертывание: вы можете перезапускать, обновлять или масштабировать производителей и потребителей совершенно независимо друг от друга.

Полиглотность: производитель на Rust может записывать данные в память, а потребитель на Python или C++ — читать их.

Производительность: стандартный межпроцессный обмен (например, TCP-сокеты или каналы) заставляет платить «налог» за изоляцию процессов в виде переключений контекста ядра и копирования памяти. Разделяемая память снимает этот налог.

Если вашей системе требуется надежная изоляция, независимые жизненные циклы компонентов или связь между несколькими языками с задержкой в ​​микросекунды, разделяемая память — это именно тот инструмент, который вам нужен.
===
Часть 1: Основы ОС, FFI и магия nix
Мы рассмотрим основы на уровне ОС, необходимые для настройки этого. Мы будем напрямую взаимодействовать с системными вызовами POSIX. Обычно это означает погружение в интерфейс внешних функций (FFI) C с использованием libc, что по своей сути небезопасно в Rust.

К счастью, крейт nix предоставляет безопасные, идиоматические обертки Rust для этих системных вызовов, избавляя нас от множества необработанных указателей.

Давайте углубимся в основной код инициализации и разберем, что делает каждая функция.

Создание сегмента разделяемой памяти: shm_create_or_open
Эта функция является основой для функции записи (производителя). Она запрашивает у операционной системы создание блока разделяемой памяти и присвоение ему соответствующего размера.
```rust
pub fn shm_create_or_open(name: &str, size: usize) -> io::Result<OwnedFd> {
  let fd = nix::sys::mman::shm_open(
    name,
    OFlag::O_CREAT | OFlag::O_RDWR,
    Mode::from_bits_truncate(0o666),
  )
  .map_err(|e| io::Error::from_raw_os_error(e as i32))?;
  //...
}
```
- shm_open против File::open: Вместо создания файла на жестком диске, shm_open создает объект разделяемой памяти POSIX (обычно расположенный в ОЗУ, например, /dev/shm в Linux).
- Флаги: O_CREAT указывает ОС создать сегмент, если он не существует, а O_RDWR предоставляет права на чтение/запись. 0o666 устанавливает стандартные права на чтение/запись для всех пользователей.
- Безопасный FFI через nix: Внутри это вызывает функцию C shm_open. Крейт nix обрабатывает границу FFI, преобразуя коды ошибок C в типы Rust Result, которые мы сопоставляем со стандартным io::Error.

Соединение читателей: shm_open_readonly и shm_open_readwrite
В то время как производитель создает память, потребители (репортеры) должны к ней подключаться.
```rust
pub fn shm_open_readonly(reader_id: usize, name: &str) -> Mmap {
  // Retry loop wait for writer fo create shared memory
  eprintln!("[reporter {}] waiting for shared memory...", reader_id);
  loop {
    if !RUNNING.load(Ordering::Relaxed) {
      panic!("[reporter {}] interrupted before connecting", reader_id);
    }
    let fd = match nix::sys::mman::shm_open(name, nix::fcntl::OFlag::O_RDWR, Mode::empty()) {
      Ok(id) => id,
      Err(_) => {
        thread::sleep(Duretion::from_millis(500));
        continue;
      }
    };
    let mmap = unsafe { Mmap::map(&fd).expect("faled to mmap shared memory")};
    return mmap;
  }
}
```
- The Retry Loop: В конфигурации «производитель-потребитель» мы не можем гарантировать, кто запустится первым. Если потребитель запускается до того, как производитель создаст разделяемую память, shm_open завершится с ошибкой. Вместо сбоя потребитель корректно приостанавливает работу на 500 мс и пытается снова.
- Проверка RUNNING: Внутри бесконечного цикла мы постоянно проверяем глобальный атомарный флаг. Если пользователь нажимает Ctrl+C, пока потребитель ожидает производителя, это предотвращает бесконечное зависание потока.
- Небезопасный Mmap: Получив дескриптор файла (fd), мы передаем его крейту memmap2. Обратите внимание, что Mmap::map — это небезопасный блок Rust. Почему? Потому что мы отображаем память, которая находится вне проверки заимствований Rust. Теоретически другой процесс может усечь или изменить эту память, нарушая строгие гарантии безопасности памяти Rust. Мы говорим компилятору: «Доверься мне, я позабочусь о синхронизации».

Плавное завершение работы: install_signal_handler

Когда вы думаете о межпроцессном взаимодействии (IPC) на основе разделяемой памяти, вы можете заметить, что оно ставит под сомнение распространенное предположение: когда процесс завершается, его память очищается. Когда несколько процессов работают вместе в одном сегменте разделяемой памяти, завершение одного процесса не должно приводить к тому, что другие внезапно начнут указывать на недействительную память.

Но если память остается после завершения процесса, возникает новая проблема: что в конечном итоге очищает память? Поскольку ОС не сделает это за вас, плавное завершение работы не является необязательным; оно обязательно при работе с IPC на разделяемой памяти. IPC:
```rust
/// Registers handlers for 'SIGINT' (Ctrl + C) and 'SIGTERM' that set the global
/// ['RUNNING'] flag to 'false', allowing main loops to exit gracefully.
///
/// Most be called once at startup, before any loop that checks 'RUNNING'.
put fn install_signal_handler() {
  unsafe {
    signal_hook::low_level::register(signal_hook::consts::SIGINT, || {
      RUNNING.store(false, Ordering::SeqCst);
    })
    .expect("failed to register SIGINT handler");

    signal_hook::low_level::register(signal_hook::consts::SIGTERM, || {
      RUNNING.store(false, Ordering::SeqCst);
    })
    .expect("failed to register SIGTERM handler");
  }
}
```
Приведённый выше код, по сути, говорит о том, что при поступлении сигналов (SIGINT и SIGTERM в данном контексте) нужно перевести глобальный флаг RUNNING в значение false. Это останавливает основной цикл, но память всё равно не удаляется.

Теперь ваша задача — обработать фактическую очистку. В Rust идиоматическое место для реализации кода очистки — это реализация трейта Drop.

В этом проекте я реализовал механизм «счётчика ссылок». По мере подключения и отключения процессов этот счётчик обновляется. Когда процесс выходит из области видимости, его реализация Drop проверяет счётчик, и если это последний оставшийся процесс, он отменяет привязку к разделяемой памяти:
```rust
impl Drop for ShmRingBufferLayout {
  fn drop(&mut self) {
    //Decrement attached_count; detach() returns the *previous* value.
    //If it was 1, we were the last process - fafe to unlink the segment.
    if self.detach() == 1 {
      let _ = shm_unlink(SHM_NAMA);
    }
  }
}
```
Теперь мы успешно заложили основу.
