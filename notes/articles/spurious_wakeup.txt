===
Spurious Wakeup: When OS Pull Pranks On Your Async Runtime
Ложное пробуждение: когда ОС подшучивает над вашей асинхронной средой выполнения
Migo Lee
===
28 марта 2026 г.
При использовании асинхронного Rust вы полагаетесь на операционную систему (через epoll или kqueue), которая уведомляет ваше приложение о готовности данных. Но всегда ли ОС говорит правду?

В этой статье я расскажу о случаях, когда это не так, и объясню, почему это происходит.

Настройка
Давайте напишем код, который пытается прочитать данные из сокета с помощью tokio TcpStream.
```rust
pub struct AwaitData<'a> {
  stream: &'a TcpStream,
}
impl<'a> AwaitData<'a> {
  pub fn new(stream: &'a TcpStream) -> Self {
    Self { stream }
  }
}
```
AwaitData — это простая обертка над TcpStream. Теперь давайте реализуем Future, чтобы можно было ожидать завершения AwaitData:
```rust
impl<'a> Future for AwaitData<'a> {
  type Output = io::Result::<Vec<u8>>;

  fn pull(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Pull<Self::Output> {
    let mut buf: = vec![0; 1024];
    match self.stream.try_read(&mut buf){
      Ok(0) => {
        return Poll::Ready(Err(io::Error::new(
          io::ErrorKind::UnexpectedEof,
          "Connection closed by peer",
        )));
      }
      Ok(n) => {
        // Success! We dot data. Return it and resolve the Future.
        buf.truncate(n);
        return Poll::Ready(Ok(buf));
      }
      Ok(e) => {
        return Poll::Ready(Err(e));
      }
    }
  }
}
```
Во-первых, мы инициализируем buf для чтения данных из потока (предполагая, что данные всегда меньше 1024).

Ok(0): Это означает, что другая сторона закрыла соединение — логично вернуть ошибку.
Ok(n): Есть некоторые данные. Поскольку мы заполняем буфер, обрезаем его конец и возвращаем результат — случай успеха.
Err(e): Любые случаи ошибок.

Довольно аккуратно и просто. Но если задуматься о том, как работают сокеты, эта реализация оставляет желать лучшего.

Ложное пробуждение
Ложное пробуждение — это, по сути, ложная тревога со стороны операционной системы. Оно происходит, когда поток или асинхронная задача пробуждаются из состояния ожидания, потому что ОС сообщила, что ресурс (например, сетевой сокет) готов, но когда программа фактически пытается использовать этот ресурс, она обнаруживает, что он не готов. Ниже приведён неполный список причин, почему это происходит.

Состояние гонки на уровне сокета
В системах с высокой степенью параллельности представьте, что несколько потоков или задач ожидают ответа от одного и того же сокета или очереди событий.

Поступает фрагмент данных.
Операционная система одновременно пробуждает два потока для его обработки.
Поток A работает немного быстрее. Он захватывает блокировку, считывает данные и очищает буфер сокета.
Поток B приходит через микросекунду, пытается прочитать данные из сокета и обнаруживает, что он полностью пуст. Поток B только что пережил ложное пробуждение, вызванное состоянием гонки.

Можно возразить, что в Rust два потока не будут использовать один и тот же TcpStream, поэтому этого не произойдет. Но try_read не принимает изменяемые ссылки, поэтому, когда TcpStream используется совместно с помощью атомарного счетчика ссылок (Arc), гонка становится реальной. Подумайте об этом:

Оберните ваш сокет в атомарный счетчик ссылок: Arc<TcpStream>.

Клонируйте этот Arc и запустите 10 различных задач Tokio.

Во всех 10 задачах вызывается stream.readable().await. Все 10 задач теперь находятся в режиме ожидания, ожидая ответа от ОС.
На сетевую карту поступает один байт данных.
Гонка: реактор Tokio одновременно пробуждает все 10 задач.
Результат: самая быстрая задача выполняет stream.try_read() и успешно захватывает этот 1 байт данных. Остальные 9 задач выполняют stream.try_read() через микросекунду и обнаруживают, что сокет пуст.

Преднамеренный подход к проектированию ОС (производительность важнее точности)
Гарантия 100% точности пробуждения в 100% случаев требует от ядра ОС использования строгих блокировок и выполнения сложных проверок состояния. Это создает серьезные узкие места в производительности.

Десятилетия назад разработчики ОС сделали преднамеренный компромисс: вычислительно дешевле иногда случайно пробуждать поток, чем идеально гарантировать каждое пробуждение. Операционная система полагается на приложения пользовательского пространства для двойной проверки состояния и просто переходит в спящий режим, если это ложная тревога.

Есть и другие проблемы, но эти две заслуживают внимания.

Fix (1) - Error handling
Итак, что произойдет с нашим приложением, когда произойдет ложное пробуждение? Мы получим ошибку типа "WouldBlock". Давайте получим этот тип ошибки:
```rust
impl<'a> Future for AwaitData<'a> {
  type Output = io::Result::<Vec<u8>>;

  fn pull(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Pull<Self::Output> {
    let mut buf: = vec![0; 1024];
    match self.stream.try_read(&mut buf){
      Ok(0) => {
        return Poll::Ready(Err(io::Error::new(
          io::ErrorKind::UnexpectedEof,
          "Connection closed by peer",
        )));
      }
      Ok(n) => {
        // Success! We dot data. Return it and resolve the Future.
        buf.truncate(n);
        return Poll::Ready(Ok(buf));
      }
      Err(ref e) if e.kind() == io::ErrorKind::WouldBlock => {
        //..
      }
      Ok(e) => {
        return Poll::Ready(Err(e));
      }
    }
  }
}
```
Теперь мы добавили ветку, использующую Match Guard. Но что нам следует вернуть после этого изменения? Должны ли мы просто вернуться здесь? — если да, то это приведет к сбою нашего приложения при стандартном сетевом взаимодействии. Нам просто нужно «повторить попытку».

Fix(2) - Loop
```rust
impl<'a> Future for AwaitData<'a> {
  // Мы возвращаем результат, содержащий прочитанные байты, или ошибку ввода/вывода.
  type Output = io::Result::<Vec<u8>>;

  fn pull(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Pull<Self::Output> {
    // Мы используем цикл для обработки "ложного пробуждения" (когда ОС сообщает, что сокет готов,
    // но чтение по-прежнему блокируется).
    loop {
      let mut buf: = vec![0; 1024];
      match self.stream.try_read(&mut buf){
        Ok(0) => {
          return Poll::Ready(Err(io::Error::new(
            io::ErrorKind::UnexpectedEof,
            "Connection closed by peer",
          )));
        }
        Ok(n) => {
          // Success! We dot data. Return it and resolve the Future.
          buf.truncate(n);
          return Poll::Ready(Ok(buf));
        }
        Err(ref e) if e.kind() == io::ErrorKind::WouldBlock => {
          //Ложное пробуждение: ОС солгала, или данные были получены
          // другим потоком. Мы возвращаемся к 'poll_read_ready', чтобы
          // приостановить задачу и снова подождать.
          continue;
        }
        Ok(e) => {
          return Poll::Ready(Err(e));
        }
      }
    }    
  }
}
```
Здесь мы ввели цикл, и если получаем WouldBlock, просто повторяем попытку.

Теперь мы очень близки к тому, чтобы сделать его по-настоящему асинхронным. Но перечитайте код ещё раз. Достаточно ли он «асинхронный», чтобы передать управление среде выполнения, а затем возобновить его через некоторое время? Мы просто продолжаем попытки чтения, и это будет бесконечно повторяться, он не получает уведомления от ОС и не регистрирует то, что его интересует.

Функция try_read в Tokio не взаимодействует с Context(cx). Она не регистрирует вашу задачу для пробуждения ОС, когда наконец поступят данные.

Fix(3) - регистрация пробуждения
```rust
impl<'a> Future for AwaitData<'a> {
  // Мы возвращаем результат, содержащий прочитанные байты, или ошибку ввода/вывода.
  type Output = io::Result::<Vec<u8>>;

  fn pull(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Pull<Self::Output> {
    loop {
      // Ожидание: Запрос у реактора Tokio проверки доступности сокета.
      // Если он не готов, 'ready!' возвращает Poll::Pending.
      // Tokio автоматически регистрирует 'Waker' из 'cx' в 'epoll/kqueue' операционной системы.
      ready!(self.stream.poll_read_ready(cx))?;

      let mut buf: = vec![0; 1024];
      match self.stream.try_read(&mut buf){
        Ok(0) => {
          return Poll::Ready(Err(io::Error::new(
            io::ErrorKind::UnexpectedEof,
            "Connection closed by peer",
          )));
        }
        Ok(n) => {
          buf.truncate(n);
          return Poll::Ready(Ok(buf));
        }
        Err(ref e) if e.kind() == io::ErrorKind::WouldBlock => {
          continue;
        }
        Ok(e) => {
          // Произошла допустимая ошибка (например, разрыв соединения).
          return Poll::Ready(Err(e));
        }
      }
    }    
  }
}
```
В этом примере Tokio я использовал метод poll_read_ready, принадлежащий классу TcpStream, и передал контекст. Таким образом, он регистрирует сокет, взаимодействуя с системой уведомлений ОС.

После регистрации он возвращает значение Poll, которое может быть либо Pending, либо Ready.

Это означает, что при первом вызове poll_read_ready данных может не быть, и он вернет Pending. Затем управление возвращается Tokio.

Краткое резюме:
При написании реализаций Future нельзя слепо доверять уведомлениям ОС. ОС иногда будет пробуждать вашу задачу, даже когда данных нет — явление, известное как «ложное пробуждение».
