14. Подробнее о Cargo и Crates.io
===
Собственная настройка сборок с помощью релизных профилей
Cargo состоит из двух основных профилей: 
- dev, который Cargo исполь­зует при выполнении команды cargo build
- release, используемый Cargo при выполнении команды cargo build --release.
Настройки по умолчанию для каждого профиля, который приме­няется, когда в файле Cargo.toml нет разделов [profile.*]:
Cargo.toml
[profile.dev]
opt-level = 0
[profile.release]
opt-level = 3
* opt-level управляет числом оптимизаций, которые будут применяться к коду, в интервале от 0 до 3

Внесение полезных документационных комментариев
Пример: Документационный комментарий для функции
/// Добавляет единицу к заданному числу.

///
/// ```
/// let arg = 5;

Документационные комментарии в качестве тестов
Добавление демонстрационных блоков кода в (```) документационные комментарии помогает проиллюстрировать способы применения библиотеки и выполнение cargo test
---
Комментирование внутри элементов
//! — добавляет докумен­тацию в элемент, содержащий комментарии, по общему правилу в src/lib.rs или внутри мо­дуля для документирования упаковки или модуля в целом
Например:
//! # Моя упаковка
//!
//! `my_crate` - это коллекция утилит, предназначенная для того, чтобы сделать
//! выполнение определенных вычислений удобнее

Публикация в Crates.io
Публикация является перманентной. Версию нельзя перезаписать, а код нельзя удалить.
cargo publish
Каждая упаковка в рабочем пространстве должна быть опубликована отдельно, нужно перейти в каталог каждой упаковки и выполнить команду cargo publish для каждой упаковки
---
Публикация новой версии существующей упаковки
изменяеть значение version, указанное в файле Cargo.toml, и публиковать ее по­вторно
---
Удаление версий из Crates.io с помощью команды cargo yank
Можно запретить любым будущим проектам добавлять упаковку в качестве новой зависимости
Отозвать версию упаковки:
cargo yank --vers 1.0.1
Отменить отзыв:
cargo yank --vers 1.0.1 --undo
===
Рабочие пространства Cargo
Рабочее пространство — это набор пакетов, которые совместно используют один и тот же файл Cargo.lock и выходной каталог. 
Пример programs/140/add:
Pабочее пространство, содержащее двоичный файл и две библиотеки. 
Двоичный файл, который обеспечит главную функциональность, зависит от двух библиотек. 
Первая библиотека будет предоставлять функ­цию add_one, а вторая — функцию add_two. 
Эти три упаковки части од­ного рабочего пространства.
Шаги для создания:
1. mkdir add
2. cd add
3. touch Cargo.toml
[workspace]
members = [
  "adder",
]
4. cargo new adder //в каталоге add
5. cargo build
* даже если попробовать выполнить команду cargo build изнутри каталога adder, то скомпилированные артефакты все равно оказались бы в add/target, а не в add/adder/target
6. изменение верхнего Cargo.toml
[workspace]
members = [
  "adder",
  "add-one",
]
7. 
8. add-one/src/lib.rs
pub fn add_one(x: i32) -> i32 {
  x + 1
}
9. adder/Cargo.toml
[dependencies]
add-one = { path = "../add-one" }
10. adder/src/main.rs
use add_one;
...
11. cargo build
12. чтобы выполнить двоичную упаковку из каталога add:
cargo run -p adder
13. зависимости добавляются в каждую упаковку, например rand = "0.3.14" и в adder и в add-one
14. дщбавление теста add-one/src/lib.rs
#[cfg(test)] ...
cargo test
или для конкретной упаковки
cargo test -p add-one
